## 1. О компании

**ShopWave** — растущий e-commerce ритейлер (средний бизнес, ~500 сотрудников), специализирующийся на продаже электроники: смартфоны, ноутбуки, наушники, игровые консоли, аксессуары. Компания работает в сегменте B2C через два канала:

- **Интернет-магазин** (shopwave.ru) — веб-витрина с каталогом, корзиной, личным кабинетом
- **Мобильное приложение** (iOS / Android) — повторяет функциональность сайта + push-уведомления

Компания основана 5 лет назад как стартап. За это время выросла с 3 до 500 сотрудников. Сейчас ShopWave находится в фазе активного масштабирования.

### Стратегические инициативы на ближайшие 12 месяцев

1. **Программа лояльности** — бонусные баллы за покупки, уровни клиентов (Bronze/Silver/Gold/Platinum), персональные скидки
2. **Выход на маркетплейсы** — интеграция с Ozon и Wildberries (товары, заказы, остатки в реальном времени)
3. **Собственная служба доставки** — отказ от внешнего подрядчика, свой автопарк и курьеры
4. **Аналитика на основе данных** — витрина данных для product-менеджеров и маркетинга (воронка продаж, когортный анализ, RFM-сегментация)

**Бизнес-цель:** за 12 месяцев выйти на технологический уровень, позволяющий рост 3x (оборот, ассортимент, клиентская база).

## 2. Текущий IT-ландшафт (AS-IS)

ShopWave выросла из стартапа — и IT-ландшафт отражает эту историю. Это классический «зоопарк»: монолит, кустарные решения и ручные процессы.

### Системы AS-IS

| Система | Тип | Что делает | Технологии | Болевые точки |
|---------|-----|-----------|-----------|--------------|
| **ShopWave Web** | SoE | Интернет-магазин: каталог, корзина, оформление заказа, ЛК клиента | PHP-монолит (самописный), MySQL | 7 лет легаси, тесты отсутствуют, деплой раз в 2 недели вручную |
| **Admin Panel** | SoE | Админка для менеджеров: управление товарами, заказами, акциями | Часть PHP-монолита | Интерфейс 2016 года, нет валидации данных, частые ошибки |
| **ShopCRM** | SoR | Клиенты, заказы, история коммуникаций | Самописная CRM на Python/Django | Нет API, интеграция через прямые запросы к БД |
| **Excel-отчётность** | SoI | Финансовые отчёты, складская аналитика | Excel-файлы на общем сетевом диске | Ручной сбор данных, ошибки копипасты, задержка 3-5 дней |
| **Payment GW** | Внешний | Приём платежей (карты, СБП) | Сторонний сервис (CloudPayments) | Стабильно |
| **Email Service** | Внешний | Транзакционные письма, рассылки | Сторонний сервис (UniSender) | Стабильно |

### Потоки данных AS-IS (упрощённо)

```
Покупатель → ShopWave Web → Payment GW → ShopWave Web → ShopCRM
                                                              ↓
Менеджер → Admin Panel → ShopWave Web (одна БД) → Excel-отчётность (руками)
                                                              ↓
Маркетплейсы ← товары ← Admin Panel (ручная выгрузка CSV) ← Менеджер
```

### Ключевые проблемы AS-IS

1. **Единая БД монолита** — любое изменение в одной фиче может сломать другую
2. **Нет API у CRM** — интеграции делаются прямыми запросами к БД, нет контрактов
3. **Ручная выгрузка на маркетплейсы** — менеджер раз в день выгружает CSV, частые расхождения остатков
4. **Excel-отчётность** — задержка 3-5 дней, бизнес принимает решения вслепую
5. **Нет версионирования** — код хранится на FTP-сервере, Git не используется
6. **Нет автоматизации тестирования** — каждый релиз — это риск

## 3. Целевой IT-ландшафт (TO-BE)

Видение на 12 месяцев: модульная архитектура, разделение ответственности, автоматизация.

### Целевые системы

| Система | Назначение | Архитектурный стиль |
|---------|-----------|-------------------|
| **Catalog Service** | Управление товарами, категориями, характеристиками | Микросервис (REST API) |
| **Order Service** | Оформление и жизненный цикл заказа | Микросервис (REST + события) |
| **Loyalty Service** | Программа лояльности: баллы, уровни, скидки | Микросервис (REST API) |
| **Delivery Service** | Собственная служба доставки: маршруты, курьеры, статусы | Микросервис (REST + события) |
| **Marketplace Adapter** | Адаптер для Ozon/Wildberries: товары, заказы, остатки | Микросервис (REST + batch) |
| **Customer Profile Service** | Единый профиль клиента (выделяется из CRM) | Микросервис (REST API) |
| **Notification Service** | Уведомления: email, SMS, push | Микросервис (события) |
| **BI Platform** | Аналитика: витрины данных, дашборды | OLAP + DWH |
| **API Gateway** | Единая точка входа для клиентов и партнёров | API Gateway (Kong / NGINX) |

### Ключевые интеграционные потоки TO-BE

```
Мобильное приложение ──┐
                       ├── API Gateway ──┬── Catalog Service
Интернет-магазин ──────┘                 ├── Order Service
                                          ├── Loyalty Service
                                          ├── Delivery Service
                                          └── Customer Profile Service

Order Service ──(Kafka: order.created)──▶ Loyalty Service (начисление баллов)
Order Service ──(Kafka: order.paid)────▶ Delivery Service (создание доставки)
Order Service ──(Kafka: order.*)───────▶ BI Platform (стриминг аналитики)

Marketplace Adapter ──(batch, раз в 15 мин)──▶ Ozon API
Marketplace Adapter ──(batch, раз в 15 мин)──▶ Wildberries API
```

## 4. Доменный глоссарий

Единая терминология, которую используют все авторы контента и студенты.

| Термин | Определение | Пример / Контекст |
|--------|------------|-------------------|
| **Клиент** | Зарегистрированный пользователь с профилем и историей заказов | Иван Петров, Gold-уровень, 12 заказов |
| **Товар** | Единица каталога с артикулом, ценой и остатком | iPhone 15 Pro 256GB, арт. IP15P-256-BLK |
| **Заказ** | Оформленная покупка одного или нескольких товаров | Заказ №12345 на сумму 89 990 ₽ |
| **Позиция заказа (OrderItem)** | Конкретный товар внутри заказа с количеством и ценой | iPhone 15 Pro ×1, чехол ×2 |
| **Бонусный балл** | Единица валюты лояльности, начисляется за покупки | 1 балл = 1 рубль, начисляется 5% от суммы заказа |
| **Уровень лояльности** | Статус клиента: Bronze, Silver, Gold, Platinum | Gold: от 50 000 ₽ общих покупок, кэшбэк 7% |
| **Кэшбэк** | Процент от суммы заказа, начисляемый бонусными баллами | Gold-клиент получает 7% от заказа баллами |
| **Маркетплейс** | Внешняя торговая площадка: Ozon, Wildberries | Товары ShopWave продаются через витрину Ozon |
| **Остаток** | Количество единиц товара на складе, доступное для продажи | iPhone 15 Pro: 23 шт на складе, 5 зарезервировано |
| **Доставка** | Процесс передачи заказа клиенту через собственную службу | Курьер №7, интервал 14:00-16:00 |
| **Bounded Context** | Логическая граница домена со своей моделью данных и языком | Контекст «Заказы» не зависит от контекста «Доставка» |
| **Витрина данных** | Агрегированные данные для аналитики (денормализованные) | Витрина «Продажи по дням» для дашборда |

## 5. Роль студента-аналитика по модулям

Студент — **единственный системный аналитик**, нанятый в ShopWave для технологической трансформации.

| Модуль | Этап работы над ShopWave | Что делает студент |
|--------|--------------------------|-------------------|
| **M1** | **Аудит текущего состояния** | Рисует AS-IS диаграмму IT-ландшафта ShopWave: все 6 систем, потоки данных, болевые точки. Описывает текущий SDLC-процесс (Waterfall), предлагает переход на Scrum для конкретных команд. Настраивает Jira-проект «ShopWave Transformation» и Confluence-space для документации. |
| **M2** | **Целевая архитектура** | Проектирует TO-BE архитектуру: выделяет bounded contexts (Заказы, Каталог, Лояльность, Доставка, Маркетплейсы, Клиенты), выбирает архитектурный стиль для каждого (микросервисы vs модульный монолит), проектирует интеграционные потоки (REST для синхронных запросов, Kafka для событий заказов, batch для выгрузки на маркетплейсы), пишет ADR на ключевые решения. |
| **M3** | **Проектирование данных** | Проектирует ER-диаграмму для контекста «Заказы + Лояльность» (6+ сущностей: Customer, Order, OrderItem, Product, LoyaltyAccount, BonusTransaction). Нормализует до 3NF. Пишет DDL для PostgreSQL. Генерирует 50 000+ строк тестовых данных. Пишет аналитические SQL-запросы: топ-10 клиентов по сумме покупок, товары которые часто покупают вместе, динамика начисления баллов по месяцам. |
| **M4** | **Разработка требований** | Проводит «интервью» с Product Owner ShopWave (роль играет модератор) по фиче «Программа лояльности». Документирует 3-5 User Stories с Acceptance Criteria. Моделирует BPMN-процесс «Начисление бонусных баллов»: от события «заказ оплачен» до зачисления баллов на счёт клиента с учётом уровня лояльности. Приоритизирует бэклог: MoSCoW (MVP: начисление баллов, Must; реферальная программа — Won't) и RICE-оценки. |
| **M5** | **Валидация решения** | Пишет 15 тест-кейсов для API Loyalty Service: начисление баллов, проверка уровня, списание баллов, граничные значения (0, -1, MAX_INT). Разрабатывает UAT-план и чек-лист для бизнес-пользователей ShopWave. Строит traceability matrix (User Story → AC → Test Case → Status). Тестирует API Loyalty Service через Postman-коллекцию с автоматическими проверками. |
| **M6** | **Capstone: защита проекта** | Собирает все артефакты M2–M5 в итоговый проектный документ. Настраивает Jira-доску с эпиками для трансформации ShopWave. Готовит презентацию (10 слайдов) для «CTO ShopWave» (роль играет ментор): текущее состояние → целевая архитектура → ключевые решения → риски → roadmap. Защищает архитектурные, data- и requirements-решения в формате Q&A. |

Конкретные фичи, которые фигурируют в практических заданиях, лекциях и статьях.

| Фича | Контекстный домен | Используется в модулях |

| **Каталог товаров** | Catalog | M2 (API Design), M3 (ERD) |

## 7. Правила для авторов контента

При создании лекций, статей, практических заданий и тестов:

1. **Используйте термины из глоссария.** Не придумывайте новые названия для одних и тех же сущностей.
2. **Ссылайтесь на ShopWave явно.** Не «вымышленная e-commerce компания», а «в ShopWave».
3. **Держитесь в рамках легенды.** Если в легенде сказано, что текущий стек — PHP-монолит, не придумывайте, что там уже есть микросервисы.
4. **Используйте конкретные цифры.** Не «много заказов», а «~5 000 заказов в день».
5. **Показывайте связь с другими модулями.** Если вы пишете статью для M2, добавьте фразу «В M3 вы спроектируете схему базы данных для этого контекста».
6. **Сохраняйте роль студента.** Студент — единственный системный аналитик в ShopWave. Не вводите других аналитиков в легенду без необходимости.
